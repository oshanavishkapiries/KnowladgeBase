---
name: study_guide_generator
description: Generate deep, large-scale, and highly detailed software engineering learning documents with practical examples, case studies, edge cases, book references, and visual diagrams/Mermaid schemas.
---

When the user asks you to generate a study guide, explain a technical topic in depth, or create a learning document, apply this skill. Follow these instructions to produce high-quality, comprehensive learning resources:

### 1. Information Gathering & Synthesis
- **Web Research**: Perform a web search on the given topic to gather the latest industry standards, tools, and updates.
- **Filter and Curate**: Filter out noise and irrelevant search results, extracting only the most accurate and pertinent information.
- **Agent Knowledge Integration**: Synthesize the gathered facts with your own deep knowledge of software engineering principles, system design patterns, and programming paradigms.

### 2. Theoretical Grounding & Literature References
- **Classic Literature**: Integrate insights and principles from seminal software engineering books where relevant. Cite specific books such as:
  - *Designing Data-Intensive Applications* (Martin Kleppmann)
  - *Clean Code* (Robert C. Martin)
  - *The Pragmatic Programmer* (Andrew Hunt & David Thomas)
  - *Patterns of Enterprise Application Architecture* (Martin Fowler)
  - *Refactoring* (Martin Fowler)
  - *Domain-Driven Design* (Eric Evans)
- Explain how these classic concepts apply to modern implementations of the topic.

### 3. Length, Depth, and Scope Constraints
- **Large-Scale & In-Depth**: Do not write short summaries or brief 1-2 paragraph overviews. Force yourself to generate a long, highly detailed, and exhaustive document.
- Write multiple structured sections covering core concepts, internal mechanisms, architectural designs, trade-offs, and future outlook.

### 4. Pedagogical Guidelines (Simple & Visual)
- **Simple English**: Use plain, clear, and direct English. Avoid overly dense jargon without first defining it in simple terms.
- **Visual Diagrams**: Always include Mermaid diagrams to help visual learners. Use sequence diagrams, flowchart paths, system architecture schemas, or state transitions to explain complex interactions.
- **Visual Metaphors**: Use tables and comparison lists to structure differences (e.g., Pros vs Cons, Latency vs Throughput).

### 5. Practical & Real-World Context
- **Practical Examples**: Provide concrete, runnable-style code snippets or step-by-step configuration files instead of purely abstract pseudocode.
- **Case Studies**: Detail how major tech companies (e.g., Google, Netflix, Amazon, Meta, Uber) solved or implemented this specific technology or pattern at scale.
- **Edge Cases & Failure Modes**: Devote a dedicated section to edge cases, potential bugs, latency spikes, failure models (e.g., split-brain, network partitions, race conditions), and how to mitigate them.
